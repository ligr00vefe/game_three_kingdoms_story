# 관우 캐릭터 스프라이트 — 기능 명세 (프롬프트 아님)

최종 개정: 2026-07-12
목적: **이미지 생성은 다른 AI/툴이 담당**한다. 이 문서는 "무엇이 필요하고, 왜 필요하고, 게임의 어느 기능에 어떻게 쓰이는지"를 코드 기준으로 정확히 정의한다. 여기 적힌 내용을 근거로 프롬프트를 작성하거나 사람이 직접 그려도 된다.
결과물을 받으면 이 문서 기준으로 **manifest.json 등록 + 코드 연동은 내가 처리**한다.

이전 문서 `AI_CHARACTER_PROMPTS.md`(프롬프트 문구 시도)는 이 문서로 대체됐다.

---

## 1. 이 캐릭터가 게임에서 하는 일 (배경)

- 장르: 옆스크롤 액션 RPG. 플레이어가 조작하는 캐릭터는 관우 1명뿐(추후 캐릭터 추가 예정, 지금은 범위 아님).
- 성장 방식: 장비 교체 없음(장비 시스템은 숨김 처리됨). 대신 **레벨이 오르면 외형이 통째로 바뀐다** — "직책" 5단계.
- 지금 실제로 필요한 산출물은 **직책 1단계(무명소졸)** 하나뿐. 나머지 4단계는 나중.

## 2. 디자인 기준 (텍스트 재서술 금지)

- 디자인의 유일한 기준은 **사용자가 최근에 확정한 참조 이미지 자체**다. 헤어스타일·수염·의상·무기 색상 등을 텍스트로 다시 설명하지 말고, **그 이미지를 그대로 소스**로 사용한다(재해석·재디자인 금지).
- 이미 확보된 것: 정면/옆/후면 "마스터 이미지". 이걸 소스로 게임용 뷰를 생성한다.
- **게임용 뷰는 메이플스토리 방식의 "우정면"(3/4 뷰)이다** — 완전한 측면(프로필)이 아니라, 몸통·얼굴이 살짝 정면을 향한 채 시선과 이동 방향은 오른쪽인 각도. 메이플스토리 캐릭터가 서 있을 때의 그 각도를 그대로 따른다. 완전 측면과 완전 정면은 둘 다 불가.
- **좌/우는 각각 별도 스프라이트로 제작한다 — 코드 반전(setFlipX) 사용 금지** (2026-07-12 확정: 반전하면 무기 잡는 손이 좌우로 바뀌어 어색함. 실제로 시도했다가 사용자 확인 후 폐기). 우정면(오른쪽 보기)과 좌정면(왼쪽 보기) **두 벌**이 필요하다.
- 예외는 climb(사다리) 하나뿐 — **뒷모습 단일 시트**로 그린다(§4 climb 참고). 뒷모습은 좌우 개념이 없어 한 벌이면 된다. 메이플과 동일한 방식.

## 3. 기술 규격 (반드시 지켜야 코드가 인식함)

| 항목 | 값 | 비고 |
|---|---|---|
| 프레임 크기 | **128×128px** | 관우 전용 규격(몬스터는 64×64, 별개) |
| 시점 | 메이플식 **우정면(3/4 뷰)** — 몸은 살짝 정면 기미, 시선·진행 방향은 오른쪽 | 완전 측면(프로필)·완전 정면 둘 다 불가. 예외: climb만 뒷모습(§4) |
| 배경 | 완전 투명 PNG | 그림자·잔상 없이 깔끔하게 |
| 파일 형식 | 동작(액션) 1개 = 이미지 파일 1개 | 여러 프레임을 가로로 이어붙인 "스트립" 1장. 서로 다른 동작을 한 파일에 섞지 않음 |
| 파일명 | `guanwu_t1_{동작명}_{r\|l}.png` — r=오른쪽, l=왼쪽 (climb만 방향 없이 `guanwu_t1_climb.png`) | 오타 나면 해당 동작만 조용히 예전 모습(대체 이미지)으로 남고 게임은 안 깨짐 |
| 저장 위치 | `frontend/public/assets/img/char/` | 내가 manifest.json에 등록 |
| 발 위치 | 모든 프레임에서 발바닥이 프레임 하단 동일 높이(±4px) | 안 맞으면 재생할 때 캐릭터가 위아래로 떨림 |
| 프레임 간 일관성 | 머리 크기, 무기 길이, 캐릭터 비율이 **모든 프레임에서 완전히 동일**해야 함(각도·자세만 변화) | 이게 안 지켜지면 재생 시 캐릭터가 커졌다 작아졌다 함(실제로 겪은 실패 사례) |

## 4. "동작(액션)"이 게임에서 뭘 의미하는지

게임 캐릭터는 상태 하나를 가지고 있고(지금 걷는 중/공격 중/맞는 중 등), 상태가 바뀔 때마다 그 상태에 맞는 스프라이트 시트가 재생된다. 아래는 **상태별로 언제 재생되고, 왜 필요하고, 무엇이 달라져야 하는지**를 게임 로직 기준으로 설명한다.

### idle (대기)
- **언제**: 캐릭터가 멈춰서 아무 입력도 없을 때. 게임 대부분의 시간 동안 보이는 기본 모습.
- **프레임 4장, 무한 반복**.
- **기능적으로 필요한 것**: "숨쉬기"처럼 살아있다는 느낌만 주면 됨 — **가슴·어깨가 미세하게 오르내리는 수준**의 변화(자세·팔다리 위치는 고정). (실제로 이 정도로 미세한 차이는 이미지 생성으로 만들기 어려워서, 마스터 이미지 1장을 그대로 쓰고 위아래로 1~2px씩 옮기는 방식으로 내가 코드/이미지 처리로 만들 계획. 이 동작은 별도 생성 불필요할 가능성 높음.)

### walk (이동)
- **언제**: 좌우 이동 키를 누르고 있을 때(지상).
- **프레임 6장, 무한 반복**.
- **티어별 모션 구분 (2026-07-12 확정)**: **T1 무명소졸은 걷기 모션으로 충분** — 현재 적용된 T1 아트 유지. **T2 이상(의병장~무신)은 달리기(run) 모션**으로 생성한다(직책이 오를수록 기동감). 파일명은 티어 무관하게 walk.
- **기능적으로 필요한 것**: 다리가 교차하는 1이동 사이클. 왼발 앞→오른발 앞이 자연스럽게 반복. 무기는 손에 들고 이동하는 자세 유지(전투 동작 아님).

### jump (점프)
- **언제**: 공중에 떠 있을 때(상승/하강 모두 이 동작 하나로 처리).
- **프레임 2장**. 반복 없음 — **코드가 속도에 따라 프레임을 직접 선택**한다(위로 움직이면 1번 프레임, 아래로 떨어지면 2번 프레임을 고정해서 보여줌. 애니메이션처럼 자동 재생되는 게 아님).
- **기능적으로 필요한 것**: 1번=상승 중(무릎 당겨 올림), 2번=하강 중(다리 뻗음). 정확히 2개의 "정지된 자세"만 있으면 됨.

### ~~dash (공중 대쉬)~~ — ❌ 별도 그림 불필요 (2026-07-12)
- 대쉬는 **점프와 동일한 모션을 재사용**한다. 점프 자세 그대로 수평 이동하며, 뒤에 잔상/스피드라인 이펙트(`onAirDash`, 코드가 처리)가 붙어 돌진으로 읽힌다.
- **jump 그림만 만들면 됨.** 대쉬 전용 아트는 제작하지 않는다.

### climb (사다리)
- **언제**: 사다리를 오르내리는 중.
- **프레임 2장, 무한 반복**(단, 멈춰있으면 재생도 멈춤 — 오르내릴 때만 프레임 진행).
- **기능적으로 필요한 것**: 손과 발을 교대로 움직이는 2프레임 사이클. **구도는 뒷모습(등이 보이는 각도)로 확정** — 메이플스토리가 사다리/줄에서 쓰는 방식과 동일하고, 사다리는 벽 쪽을 보고 타는 게 자연스럽기 때문. 12개 동작 중 유일한 예외 구도. 뒷모습은 좌우 대칭이라 반전 처리와도 충돌 없음.

### ~~sit (앉기)~~ — ❌ 기능 제거 (2026-07-12)
- 이동 구성을 **달리기 / 대쉬 / 점프**로 확정하면서 앉기 기능을 게임에서 제거했다. **그림 제작 불필요.**
- 코드는 `Player.ts`의 `SIT_ENABLED` 플래그로 보존돼 있어 차후 부활 가능.

### thrust (기본공격 - 찌르기)
- **언제**: 공격 키를 누르면 "찌르기"와 "휘두르기" 중 **50% 확률로 랜덤 선택**되는 두 가지 기본 공격 중 하나. 지상/공중 모두 사용 가능.
- **프레임 6장**, 1회 재생. 전체 재생 시간 0.35초, **재생 시작 후 0.12초 시점에 실제 타격 판정이 발생** — 프레임으로 환산하면 **6장 중 3번째 프레임에서 창끝이 최대로 뻗어 있어야** 판정과 시각이 맞음(1~2번째: 당기기 예비동작, 3번째: 최대 신전, 4~6번째: 회수).
- **기능적으로 필요한 것**: 창을 뒤로 당겼다가 앞으로 직선으로 찔러 넣는 동작. 창끝이 캐릭터 몸보다 앞으로 길게 뻗어나가야 함(공격 사거리가 김 — 창 특성).

### swing (기본공격 - 휘두르기)
- **언제**: thrust와 같은 조건(50% 확률로 이쪽이 선택).
- **프레임 6장**, 1회 재생, 타이밍은 thrust와 동일(0.35초 재생, 0.12초 시점 타격 = **3번째 프레임에서 칼날이 전방을 지나가는 순간**이어야 함. 1~2번째: 치켜들기, 3번째: 전방 통과, 4~6번째: 휘두름 마무리).
- **기능적으로 필요한 것**: 창을 머리 위로 들었다가 전방으로 크게 수평으로 휘두르는 동작.

### skill (스킬 발동)
- **언제**: 스킬 키/퀵슬롯으로 액티브 스킬 발동 시. **현재 코드는 어떤 스킬이든 이 동작 하나만 재생한다**(참마돌격이든 청룡참이든 뇌신강림이든 전부 같은 모션). 스킬별로 다른 모션을 원하면 이건 별도 개발 작업이 필요함 — 지금은 **"강력한 필살기 한 방"이라는 범용 모션 1개**만 있으면 됨.
- **프레임 8장**, 1회 재생. 전체 0.45초, **0.15초 시점에 타격 판정** — 프레임으로 환산하면 **8장 중 3~4번째 프레임이 최대 타격 자세**여야 함(1~2번째: 기 모으기 웅크림, 3~4번째: 필살 휘두름, 5~8번째: 여운/회복).
- **기능적으로 필요한 것**: 기 모으기(웅크림) → 크게 휘두르는 필살 동작 → 착지/회복. 기본공격보다 확실히 더 크고 화려해야 함(플레이어가 "필살기 쐈다"고 체감할 수 있게).

### hit (피격)
- **언제**: 몬스터 공격에 맞았을 때. (붉은 점멸, 넉백, 1초 무적은 이미 코드가 처리 — 그림은 자세만 있으면 됨)
- **프레임 2장**, 1회 재생.
- **기능적으로 필요한 것**: 움찔하는 자세 딱 2개면 충분. 여기에 공들일 필요 없음(연출은 코드가 대신 함).

### dead (사망)
- **언제**: HP 0 도달 시.
- **프레임 5장**, 1회 재생 후 마지막 프레임(쓰러진 채) 유지.
- **기능적으로 필요한 것**: 비틀거림 → 무릎 꺾임 → 옆으로 쓰러짐 → 완전히 눕는 단계적 붕괴.

### rally (사기충전) — ⚠️ 아직 게임 로직에 연결 안 됨
- **의도**: "무장의 위엄" 스킬(공격력 버프)용으로 예약된 동작 — 무기를 머리 위로 치켜들고 함성 지르는 자세.
- **프레임 4장**, 1회 재생.
- **현재 상태**: 애니메이션 시스템에는 슬롯이 준비돼 있지만, **실제로 이 동작을 트리거하는 게임 로직은 아직 구현 안 됨**(스킬 자체도 데이터만 있고 효과 미구현). 그림을 미리 만들어둬도 되지만, 당장 게임에서 재생되진 않는다는 점 참고.

## 5. 파일명·프레임 수 요약표

각 동작은 **오른쪽(_r)·왼쪽(_l) 두 파일**이 필요하다 (climb만 뒷모습 단일 파일).

|---|---|---|---|---|
| `guanwu_t1_idle_r.png` / `_l.png` | idle | 4 | 무한loop | ✅ 완료 (2026-07-12 게임 적용) |
| `guanwu_t1_walk_r.png` / `_l.png` | walk(달리기) | 6 | 무한loop | ✅ 완료 (2026-07-12 게임 적용) |
| `guanwu_t1_jump_r.png` / `_l.png` | jump (대쉬도 공용) | 2 | 코드가 프레임 직접 선택(반복 없음) | 미제작 |
| ~~dash~~ | dash | — | — | ❌ 불필요 (jump 재사용) |
| `guanwu_t1_climb.png` (뒷모습, 단일) | climb | 2 | 무한loop(움직일 때만) | ✅ 완료 (2026-07-16 게임 적용) |
| `guanwu_t1_thrust_r.png` / `_l.png` | thrust | 6 | 1회 | ✅ 완료 (2026-07-15 게임 적용) |

- 아트가 없는 동작은 게임이 **같은 방향의 idle로 자동 폴백**한다 (안 깨짐, placeholder 깜빡임 없음). 파일만 추가하면 코드 수정 없이 살아난다.
- 원본 시트는 셀 격자(예: 2×2, 3×2, 6×1) 어떤 배치든 괜찮고 배경도 투명/흰색 무관 — 코드 연동 시 내가 128×128 스트립 규격으로 재가공한다. 단 **한 시트 안에서 캐릭터 비율·크기 일관성**은 지켜져야 한다.

**제작 우선순위 제안**: swing(휘두르기)이 다음 순서 — thrust는 2026-07-15 완료되어 idle·walk·thrust로 기본 전투 골격이 섰다. climb은 2026-07-16 완료. 그다음 jump/skill/hit/dead/rally.

## 6. 결과물을 나(코드)에게 전달하는 방법

1. 생성된 시트(셀 격자 형태 그대로 OK)를 `frontend/public/assets/create/`에 저장하고 어떤 동작·방향인지 알려준다 (예: `right_thrust_demo.png`). — 실제 워크플로우(2026-07-12 idle/walk 적용 시 검증됨).
2. 내가 배경 제거 → 캐릭터 크기 통일(**높이 78px**, 발바닥을 프레임 하단 y=127에 정렬, 가로 중심 x=64) → `128×프레임수` 스트립 재가공 → `assets/img/char/` 배치까지 처리한다. manifest는 이미 전 동작이 등록돼 있어 파일만 놓이면 로드된다.
   - 높이 78px은 기존 idle/walk/attack 시트를 실측한 값(76~79)이다. 이전에 이 문서에 적혀 있던 "68px"은 실제 아트와 맞지 않아 2026-07-16 정정했다. 한 시트 안의 모든 프레임은 **동일 배율**로 축소해야 한다(프레임마다 높이를 맞추면 캐릭터가 커졌다 작아진다 — §3 참고).
3. 게임에서 실제 재생되는지(방향 전환·프레임 진행·투명 배경) 검증까지 내가 진행하고 스크린샷으로 보여준다.
